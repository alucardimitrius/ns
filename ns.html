WHOIS, nslookup, tracert, nmap — all on Windows
Windows CMD (built-in — no install needed) who.is (browser WHOIS)
How to do it — follow each step in order
  1. 1
    Open CMD. Run nslookup google.com. Screenshot the output showing the IP address resolved (e.g. 142.250.x.x).
  2. 2
    Run tracert google.com. Screenshot all the hops with their latency times. This is Windows' version of Linux's traceroute.
  3. 3

  5. 5
    Open Zenmap and scan scanme.nmap.org. Screenshot the OS detection result and service/version information shown.
Expected output
nslookup google.com → Address: 142.250.x.x tracert: 10–15 hops listed with round-trip times who.is: Registrar: MarkMonitor Inc., Created: 1997-09-15 nmap scanme.nmap.org: 22/tcp open ssh, 80/tcp open http

IPSec & Snort (IDS)
Use WSL on Windows instead of TryHackMe
WSL (Windows Subsystem for Linux) — free from Microsoft Store
💡 WSL gives you a full Linux terminal inside Windows. It's free, built by Microsoft, and takes about 5 minutes to set up.
How to do it — follow each step in order
  1. 1
    Install WSL: Open PowerShell as Administrator and run wsl --install. Restart your computer when prompted. This installs Ubuntu on your Windows machine.
  2. 2
    Open Ubuntu from the Start menu. Install IPSec tools by running: sudo apt-get install strongswan -y
  3. 3
    Run sudo ipsec restart, then run sudo ipsec statusall. Screenshot the terminal output showing the StrongSwan version and daemon status.
  4. 4
    Install Snort: Run sudo apt-get install snort -y. Then run sudo snort -V to confirm it installed. Screenshot the version banner with the Snort pig ASCII art.
  5. 5
    Run sudo snort -i eth0 -A console, then from CMD (not WSL), ping your own machine. Screenshot Snort detecting the ICMP packets live in the console.
Expected output
ipsec statusall: "Status of IKE charon daemon (StrongSwan 5.x)" snort -V: Version 2.9.x.x GRE (Build xxx) — "Snort! by Martin Roesch" Snort alert: [**] [1:408:5] ICMP Echo Request [**] {ICMP} 192.168.x.x → 192.168.x.x

Firewall Rules with iptables
Run inside WSL (Ubuntu on Windows)
WSL Ubuntu — same install as Exp 4
How to do it — follow each step in order
  1. 1
    Open WSL Ubuntu terminal. Run sudo iptables -L to list the current rules. Screenshot it — the chains should be empty at this point.
  2. 2
    Block port 80 (HTTP): Run sudo iptables -A INPUT -p tcp --dport 80 -j DROP. Then run sudo iptables -L -v to verify. Screenshot showing the new DROP rule for port 80.
  3. 3
    Allow port 443 (HTTPS): Run sudo iptables -A OUTPUT -p tcp --dport 443 -j ACCEPT. Screenshot both the INPUT and OUTPUT chains with your rules visible.
  4. 4
    Allow loopback: Run sudo iptables -A INPUT -i lo -j ACCEPT. Screenshot the full ruleset with all three rules visible together.
  5. 5
    Clean up: Run sudo iptables -F to flush all rules. Screenshot the empty state again — this shows a clear before/after comparison for your journal.
Expected output
Chain INPUT (policy ACCEPT) target prot opt source destination ACCEPT all -- anywhere anywhere (loopback) DROP tcp -- anywhere anywhere tcp dpt:http Chain OUTPUT (policy ACCEPT) ACCEPT tcp -- anywhere anywhere tcp dpt:https

Windows Command-Line Administration
100% built-in — no installs, no internet needed
Windows CMD only — everything is pre-installed
How to do it — follow each step in order
  1. 1
    Press Win + R, type cmd, and press Enter. Run systeminfo. Screenshot it — make sure the OS Name, OS Version, Total Physical Memory, and Hostname are visible.
  2. 2
    Run ipconfig /all. Screenshot the result showing your IPv4 Address, Physical (MAC) Address, DNS Servers, and Default Gateway.
  3. 3
    Run ping google.com, then run tracert google.com. Screenshot both — ping shows round-trip time, tracert shows each router hop along the route.
  4. 4
    Run netstat -abon. This shows active connections along with the process name responsible for each. Screenshot a portion of the output showing Local/Foreign addresses and PIDs.
  5. 5
    Run nslookup google.com and screenshot the DNS resolution. Then demonstrate file commands in sequence: mkdir TestFolder → cd TestFolder → echo Hello > test.txt → type test.txt → dir. Screenshot the full sequence.
Expected output
systeminfo: OS Name: Microsoft Windows 10/11 Pro ipconfig /all: IPv4: 192.168.x.x, MAC: xx-xx-xx-xx-xx-xx ping google.com: Reply from 142.250.x.x: bytes=32 time=15ms TTL=57 tracert: 1 2ms 192.168.1.1 2 25ms ... 13 42ms google.com

Wireshark Packet Capture
Install Wireshark for Windows directly
Wireshark for Windows — wireshark.org (free) Windows CMD (to generate traffic)
How to do it — follow each step in order
  1. 1
    Download and install Wireshark from wireshark.org. Launch it. Select your active interface (Wi-Fi or Ethernet). Click the blue shark fin icon to start capturing.
  2. 2
    Capture ICMP (ping) packets: Open CMD and run ping google.com -n 10. In the Wireshark filter bar, type icmp and press Enter. Screenshot the Echo Request and Echo Reply packets side by side.
  3. 3
    Capture TCP three-way handshake: Clear the filter. Open any website in your browser. Set the filter to tcp. Screenshot showing the SYN → SYN-ACK → ACK sequence. Click a packet to expand and show the Ethernet / IP / TCP layers in the detail pane.
  4. 4
    Capture HTTP (plaintext): Set the filter to http. Visit http://neverssl.com — it stays on plain HTTP. Screenshot the GET request with the Host header clearly visible in plaintext.
Expected output
ICMP filter: Echo (ping) request → 8.8.8.8, reply ← 8.8.8.8 TCP: [SYN] → [SYN, ACK] → [ACK] (three-way handshake visible) HTTP: GET / HTTP/1.1 Host: neverssl.com (plaintext readable) Packet detail pane: Frame → Ethernet → IP → TCP layers expanded

DoS Attack Simulation with Hping3
Use WSL for Hping3 + Wireshark to observe the flood
WSL (Ubuntu) Wireshark for Windows
⚠️ Target only your own loopback address (127.0.0.1) or your LAN IP — never attack external servers. This is safe, legal, and demonstrates the concept perfectly for your journal.
How to do it — follow each step in order
  1. 1
    Open WSL (Ubuntu). Install Hping3: sudo apt-get install hping3 -y
  2. 2
    Open Wireshark on Windows. Start a capture on the Loopback interface (or your Wi-Fi adapter). Leave it running.
  3. 3
    In the WSL terminal, run the flood command: sudo hping3 --flood -S -p 80 127.0.0.1. This sends thousands of TCP SYN packets per second to your own loopback. Let it run for about 5 seconds, then press Ctrl+C to stop.
  4. 4
    In Wireshark, apply the filter tcp.flags.syn == 1. Screenshot the massive spike of SYN packets all targeting port 80 — this visually proves the DoS flood.
  5. 5
    Go to Statistics → I/O Graph in Wireshark. Screenshot the graph showing the sharp packet-rate spike when the flood ran, and the sudden drop back to baseline when you stopped. This is the clearest evidence for your journal.
Expected output
Hping3: HPING 127.0.0.1 (lo): S set, 40 headers + 0 data bytes Wireshark: thousands of [SYN] packets/sec → 127.0.0.1:80 I/O Graph: sharp spike to ~50,000 packets/sec, then drops to 0 After Ctrl+C: traffic instantly returns to baseline

OWASP Web App Vulnerabilities
PortSwigger labs for all three vulnerability types
Browser + DevTools (F12) portswigger.net/web-security (free) Wireshark (for crypto failure demo)
How to do it — follow each step in order
  1. 1
    Authentication Failure: On PortSwigger, open a "Username enumeration via different responses" or "Password brute-force via response timing" lab. The lab allows unlimited login attempts. Screenshot how you crack the login and the "Solved" banner.
  2. 2
    IDOR (Broken Access Control): Open a PortSwigger Insecure Direct Object References lab. Change a URL parameter (e.g. from /my-account?id=wiener to another user's ID) to read their private data. Screenshot the unauthorized data and the solved banner.
  3. 3
    Cryptographic Failure (HTTP vs HTTPS): Open Wireshark and start capturing. Visit http://neverssl.com (plain HTTP). Apply the http filter. Screenshot showing your browser's GET request with the Host header in readable plaintext. Then visit any HTTPS site — Wireshark will show only encrypted TLS data. Screenshot both for comparison.
  4. 4
    Answer the theory questions in your journal: Unlimited login attempts → Authentication Failure · HTTP cleartext data → Cryptographic Failure.
Expected output
Auth failure lab: login brute-forced → "Congratulations, you solved the lab!" IDOR: URL param changed → another user's chat log / file accessed Wireshark HTTP: GET / HTTP/1.1 Host: neverssl.com — all plaintext readable Wireshark HTTPS: Encrypted Application Data (nothing readable)
