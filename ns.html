IPSec & Snort (IDS)
Use WSL on Windows instead of TryHackMe
▶
Tools needed
WSL (Windows Subsystem for Linux) — free from Microsoft Store
WSL gives you a full Linux terminal inside Windows. It's free, built by Microsoft, and takes about 5 minutes to install.
Practical steps
  1. Enable WSL: open PowerShell as Administrator and run wsl --install. Restart when prompted. This installs Ubuntu on your Windows machine.
  2. Open Ubuntu from the Start menu. Run sudo apt-get install strongswan -y to install IPSec tools.
  3. Run sudo ipsec restart then sudo ipsec statusall. Screenshot the terminal output showing StrongSwan version and status.
  4. For Snort: run sudo apt-get install snort -y. Then run sudo snort -V to confirm installation. Screenshot the version banner with the Snort pig ASCII art.
  5. Run sudo snort -i eth0 -A console then ping your own machine from CMD. Screenshot Snort detecting the ICMP packets in the console.
Expected output
ipsec statusall: "Status of IKE charon daemon (StrongSwan 5.x)" Snort -V: Version 2.9.x.x, "Snort! by Martin Roesch" Snort alert: [**] [1:408:5] ICMP Echo Request [**] {ICMP} 192.168.x.x → 192.168.x.x

Firewall Rules with iptables
Run inside WSL (Ubuntu on Windows)
▶
Tools needed
WSL (Ubuntu) — same install as Exp 4
Practical steps
  1. Open WSL Ubuntu terminal. Run sudo iptables -L to list current (empty) rules. Screenshot showing empty chains.
  2. Add a block rule: sudo iptables -A INPUT -p tcp --dport 80 -j DROP. Then list again with sudo iptables -L -v. Screenshot showing the new DROP rule for port 80.
  3. Add an accept rule: sudo iptables -A OUTPUT -p tcp --dport 443 -j ACCEPT. Screenshot showing both INPUT and OUTPUT chains with your rules.
  4. To show the localhost rule from the experiment: sudo iptables -A INPUT -i lo -j ACCEPT. Screenshot the full ruleset with all three rules visible.
  5. Flush all rules to clean up: sudo iptables -F. Screenshot the empty state again to show before/after.
Expected output
Chain INPUT (policy ACCEPT) target prot opt source destination ACCEPT all -- anywhere anywhere (loopback) DROP tcp -- anywhere anywhere tcp dpt:http Chain OUTPUT (policy ACCEPT) ACCEPT tcp -- anywhere anywhere tcp dpt:https

Wireshark Packet Capture
Install Wireshark for Windows directly
▶
Tools needed
Wireshark for Windows — wireshark.org (free) Windows CMD (to generate traffic)
Practical steps
  1. Download and install Wireshark from wireshark.org. Launch it. Select your active interface (Wi-Fi or Ethernet). Click the blue shark fin to start capturing.
  2. ICMP capture: Open CMD, run ping google.com -n 10. In Wireshark filter bar type icmp and press Enter. Screenshot the Echo Request and Echo Reply packets.
  3. TCP capture: Clear filter. Open any website in your browser. Filter with tcp. Screenshot showing SYN, SYN-ACK, ACK (three-way handshake). Click a packet to expand and show Ethernet/IP/TCP layers.
  4. HTTP capture: Filter with http. Visit a plain HTTP site (try http://neverssl.com — it stays HTTP). Screenshot showing GET request with Host header visible.
Expected output
ICMP filter: Echo (ping) request → 8.8.8.8, reply ← 8.8.8.8 TCP: [SYN] → [SYN, ACK] → [ACK] (three-way handshake visible) HTTP: GET / HTTP/1.1 Host: neverssl.com (plaintext visible) Packet detail pane: Frame → Ethernet → IP → TCP layers expanded

DoS Attack Simulation with Hping3
Use WSL for Hping3 + Wireshark to observe
▶
Tools needed
WSL (Ubuntu) Wireshark for Windows
Target your own loopback address (127.0.0.1) or your own LAN IP — never attack external servers. This is safe, legal, and demonstrates the concept perfectly.
Practical steps
  1. Install Hping3 in WSL: sudo apt-get install hping3 -y.
  2. Start Wireshark on Windows, capturing on the Loopback interface (or your Wi-Fi adapter).
  3. In WSL terminal, run: sudo hping3 --flood -S -p 80 127.0.0.1 to flood your own loopback with TCP SYN packets. Let it run for 5 seconds, then Ctrl+C to stop.
  4. In Wireshark, apply filter tcp.flags.syn == 1. Screenshot the massive spike of SYN packets all targeting port 80 — this visually shows the DoS flood.
  5. Screenshot the Wireshark stats (Statistics → I/O Graph) showing the packet rate spike and then the sudden drop when you stopped. This is the clearest evidence of DoS effect.
Expected output
Hping3: HPING 127.0.0.1 (lo): S set, 40 headers + 0 data bytes Wireshark: thousands of [SYN] packets/sec → 127.0.0.1:80 I/O Graph: sharp spike to ~50,000 packets/sec, then drops to 0 After Ctrl+C: traffic instantly returns to baseline

OWASP Web App Vulnerabilities
PortSwigger labs for all three vulnerability types
▶
Tools needed
Browser + DevTools (F12) portswigger.net/web-security (free) Wireshark (for crypto failure demo)
Practical steps
  1. Authentication failure: On PortSwigger, open a "Username enumeration via different responses" or "Password brute-force via response timing" lab. Screenshot how unlimited attempts are allowed and how you crack the login.
  2. IDOR (Broken Access Control): Open a PortSwigger "Insecure Direct Object References" lab. The lab has you change a URL parameter (e.g. from /my-account?id=wiener to another user's ID) to read their data. Screenshot the unauthorized data and the solved banner.
  3. Cryptographic failure: Open Wireshark, visit http://neverssl.com (plain HTTP), filter http, and screenshot showing your browser request with Host/GET headers in plaintext — demonstrating credentials/data would be exposed with no HTTPS. Compare this with visiting an HTTPS site where Wireshark shows encrypted TLS data instead.
  4. For Q1 and Q2 from the experiment theory section — write the answers in your journal: unlimited login = Authentication Failure; HTTP cleartext = Cryptographic Failure.
Expected output
Auth failure lab: login brute-forced → "Congratulations, you solved the lab!" IDOR: URL param changed → another user's chat log / file accessed Wireshark HTTP: GET / HTTP/1.1, Host: neverssl.com — all readable plaintext Wireshark HTTPS: Encrypted Application Data (nothing readable)
